# CLAUDE.md — working notes for Claude Code

Project: **RERA Easy — Maharashtra real-estate market intelligence** built from a single
MahaRERA Excel export. Read `README.md` first for the folder map and rebuild commands.

## Architecture (data flow)

```
data/Project RERA stats - Raw Data.xlsx
        │  src/prep.py   (clean + dictionary-encode into columnar arrays)
        ▼
data/data.json ──► src/build_dashboard.py ─► RERA Market Analysis Dashboard.html
        │                (injects DATA into src/dashboard_template.html)
        │  src/build_stats.py
        ▼
data/stats.json ─► src/build_doc.js ───────► RERA Easy - Maharashtra Market Analysis.docx
```

`src/developer_snapshot.html` is a hand-authored static mailer (numbers are baked in from
`stats.json`); it is not generated by a script.

## Data model (`data/data.json`)

Columnar + dictionary-encoded to keep the file ~3 MB so it can be inlined in the HTML.

- `n` — row count (54,281).
- `dict` — string tables: `ptype, pstatus, cstatus, div, dist, tal, vil, pin, bank`.
- `cols` — parallel arrays, one entry per project:
  - `yr, mo` registration year/month
  - `pt, ps, cs` codes into ptype / pstatus / cstatus
  - `dv, di, ta, vi, pn` geography codes (Division, District, Taluka, Village, Pincode) — a
    strict hierarchy; Pincode is the deepest drill level (`GEO`/`GEOMAX` in the template)
  - `bk` bank code, `cl` RERA-Easy clientele (0/1)
  - `un` units, `la` land area, `ca` carpet area, `bu` built-up, `pf` permissible FSI,
    `cp` complaints, `dy` avg days to complete
  - `ca` and `dy` may be `null` (see cleaning rules in README). Sum/average code must skip nulls.

Status semantics: `cs` 0=Active, 1=Lapsed, 2=Completed.

## The dashboard (`src/dashboard_template.html`)

- **One self-contained file. NO external libraries or CDNs** — an earlier version pulled Chart.js
  from a CDN and rendered blank in the client's offline/locked-down environment. All charts are
  hand-drawn inline **SVG** (`donut`, `lineChart`, `areaBar`) and all data is inlined at
  `/*DATA_PLACEHOLDER*/`. Keep it that way.
- Everything recomputes client-side from `DATA` via `aggregate()` → `enrich()` → `render()`.
- `render()` is wrapped in try/catch and surfaces errors into `#errbar`; keep that.
- Drill state = `path[]` (geo). `viewDim` ∈ `geo|pt|ps|bk` drives what the cards/table group by.
  Clicking a non-geo card applies it as a filter then returns to `geo` (see `clickItem`).
- Tooltips: any SVG element with a `data-tip` attribute shows the floating `#tip` div
  (global delegated mouseover/mousemove/mouseout). Donut slices instead update the center label.
- Brand theme: `--yellow:#F5C518`, `--band:#171717`, white cards on `#ececec`,
  green `#23a55a` / red `#e5484d`. Match this everywhere (dashboard, report, mailer).

## Build / edit gotchas

- After editing `src/dashboard_template.html`, always re-run `python src/build_dashboard.py`
  and re-check syntax: `node --check` on the injected `<script>` block, or open in a browser.
- The template's JS uses **string concatenation only (no backtick template literals)** so it
  survives shell here-doc authoring. Keep new code the same to avoid `$`/backtick escaping bugs.
- `build_doc.js` needs `node_modules/docx` (`npm install docx`). Validate the output with the
  docx skill's `validate.py` if available.
- Indian number format: `fmt()` renders Cr / L; keep for lakh/crore-friendly display.

## Common requests & where to change them

- **New KPI / metric** → `render()` `kpiDefs` (dashboard) and/or `build_doc.js` key-metrics table.
- **New chart** → add an SVG builder like `areaBar`, call it in `render()`, give rects `data-tip`.
- **Report wording / sections** → `buildReport()` in the template (on-screen report) and
  `src/build_doc.js` (Word). Keep the two in sync.
- **Mailer copy / target city** → edit `src/developer_snapshot.html` (static numbers).
- **Refresh data** → replace the xlsx, re-run the four builds.

## Verification habit

Before handing back a dashboard change, confirm totals against the pipeline output and that
`#errbar` stays hidden. Counts and year references are now **derived from the data** (see
`build_stats.py` year metadata and `YR_MIN/YR_MAX/MW_*` in the template), so baselines shift when
the export is refreshed. Current file: `prep.py` prints the project count (**53,536** after the
carpet/land/built-up error rows are deleted; ~38.6% completed, ~26.3% lapsed). A new same-structure
export should rebuild cleanly with no manual edits (footer count/year, launch/YTD notes, momentum
window all auto-update).
